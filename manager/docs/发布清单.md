## 2. 质量门槛（每次发布必须全绿）

在 `manager/wow-addon-manager/` 下执行（推荐用 Makefile 聚合入口）：

```bash
make ci          # 一次跑全部门槛：依赖 + 前端 typecheck/build + 后端 clippy/test
# 或分步：
make fe          # 前端：typecheck + build
make be          # 后端：clippy(-D warnings) + test
```

等价的原始命令：

```bash
# 前端
pnpm typecheck        # tsc --noEmit
pnpm build            # tsc + vite build

# 后端（src-tauri/）
cargo build
cargo test            # 单元 + 集成
cargo clippy --all-targets -- -D warnings
```

> GitHub Actions `ci.yml` 在每次 push/PR 时于 Ubuntu 上执行 `make ci`，与本地门槛一致。

最近一次结果（macOS 26.2 / arm64）：

| 门槛 | 结果 |
|---|---|
| `cargo clippy -D warnings` | ✅ 无警告 |
| `cargo test` | ✅ 78 单元测试 + 16 集成测试全通过 |
| `pnpm typecheck` | ✅ 通过 |
| `pnpm build` | ✅ 产物生成 |

集成测试覆盖（`src-tauri/tests/`）：
- `platform_fixture` —— 默认目录扫描 / flavor 识别。
- `scanner_fixture` —— `AddOns` 插件扫描。
- `installer_pipeline`（6 项）—— zip 标准/嵌套/多插件/无效/无 toc/更新备份。
- `config_snapshot` —— WTF 快照创建→落库→恢复→删除。
- `profile_apply`（2 项）—— Profile 启停目录并同步 DB / 空 Profile 禁用全部。
- `provider_pipeline`（3 项）—— LocalZip 与 GitHub 桩下载→安装、分发与未知源。
- `db_persistence` —— 文件库持久化与迁移幂等。
- `e2e_golden_path` —— **黄金链路端到端**（见 §3）。

## 4. 打包产物

在 `manager/wow-addon-manager/` 下执行 `make bundle`（等价 `pnpm tauri build`）。

| 平台 | 产物 | 状态 |
|---|---|---|
| macOS (aarch64) | `wow-addon-manager.app`、`wow-addon-manager_0.1.0_aarch64.dmg`（约 5.8 MB） | ✅ 已生成于 `src-tauri/target/release/bundle/` |
| macOS (x86_64) | `.app` / `.dmg` | ⏳ 需在 Intel 机或 `make bundle-mac`（通用包）构建 |
| Windows 11 | `.msi` / `.exe`（NSIS） | ⏳ 需在 Windows 环境执行 `make bundle` |

> 打包未做代码签名 / 公证；分发前如需去除 Gatekeeper 拦截，需补充签名与 notarization。

### CI/CD 自动发布

推送 `v*` 版本 tag（或手动触发 `release.yml`）即在 macOS 与 Windows 上并行打包并上传产物到 GitHub Release：

```bash
git tag v0.1.0
git push origin v0.1.0     # 触发 .github/workflows/release.yml
```

- CI 门槛：`.github/workflows/ci.yml`（push/PR → Ubuntu 跑 `make ci`）。
- 发布打包：`.github/workflows/release.yml`（tag `v*` → macos-latest + windows-latest 各 `make bundle` → 汇总到 Release）。
- 工作流只作薄触发层，命令收敛在 `Makefile`，本地与 CI 完全一致。
