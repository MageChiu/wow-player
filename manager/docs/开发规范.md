# 魔兽世界插件管理器 · 开发规范

> 适用范围：`manager/wow-addon-manager/` 下的全部代码与 `manager/docs/` 下的文档。
> 关联文档：[设计规划](../wow-addon-manager-agent-dev-plan.md) ｜ [开发任务](./开发任务.md)
> 本规范对所有开发者与 Agent **强制生效**。设计规划中的架构原则（§4）与跨 Agent 约束（§20）在本规范中细化为可执行条款。

---

## 0. 总原则

1. **终态架构，一次到位**：按设计规划的分层与模块边界实现，不做临时表、临时接口。
2. **模块边界不可越权**：每一层只依赖下一层，禁止跨层直接调用（UI 不碰文件系统，业务层不碰平台判断）。
3. **破坏性操作可回滚**：任何删除/覆盖/替换用户目录的操作，必须先备份、失败可恢复。
4. **错误结构化**：所有对外错误都是 `AppError`，前端据 `code` 决策。
5. **自包含**：manager 的代码与文档全部在 `manager/` 内，不外溢到仓库根或其它子项目。

## 2. 模块边界与协作约束（强制）

对应设计规划 §20，逐条给出可检查的判定。

| 编号 | 约束 | 判定方式 |
|---|---|---|
| B1 | 改公共接口必须同步类型定义 | Rust `domain` 变更的 PR 必须同时改 `src/types/*`，否则拒绝合并 |
| B2 | 不得绕过 `AppError` | command 返回类型统一 `Result<T, AppError>`；grep 无裸 `Result<_, String>` 越界 |
| B3 | 前端不得拼接本地路径 | `src/` 中禁止出现 `path.join`/字符串拼接系统路径；路径只来自后端返回值 |
| B4 | 不得直接覆盖用户目录 | 安装/恢复/应用 Profile 前必须调用备份逻辑；executor 有对应 rollback 分支 |
| B5 | Provider 不得含安装逻辑 | `providers/` 内禁止出现解压/写库/备份/移动 AddOns 的代码 |
| B6 | 平台判断只在 Platform Adapter | `services/ scanner/ installer/ providers/` 中禁止 `cfg!(windows)`/`if macos` 等分支 |
| B7 | 文件系统模块必须带 fixture 测试 | 涉及 fs 的模块 PR 必须含 `tests/fixtures/` 用例 |
| B8 | 破坏性操作必须留痕 | 安装写 `install_history`；快照写 `metadata.json` + `config_snapshots` |

**分层依赖方向（只能自上而下）**

```text
UI(src) → Command → Service → Domain
                       ↓
        Infra / Scanner / Installer / Providers → Platform Adapter
```

- 前端只能通过 `src/services/*Api.ts` 调用后端，禁止在组件里直接 `invoke`。
- Command 层只做 DTO 校验与调度，不写业务逻辑。
- Service 层编排业务，通过 repository/infra 落地，通过 Platform Adapter 取路径。

## 3. 错误模型规范（强制）

- 唯一对外错误类型：`AppError { code, message, detail?, recoverable }`（设计规划 §8）。
- 错误码只能用 `AppErrorCode` 枚举，序列化 `snake_case`。
- **不允许** `unwrap()` / `expect()` / `panic!` 出现在 command、service、installer、provider 的正常路径（测试代码除外）。
- 内部错误（`std::io::Error`、`rusqlite::Error`、网络错误等）必须在边界处映射为合适的 `AppErrorCode`：
  - 文件不可写 → `PermissionDenied`
  - zip 损坏 → `InvalidZipFile`
  - 未识别插件目录 → `NoAddonFolderDetected`
  - DB 异常 → `DatabaseError`
  - Provider/网络 → `ProviderError` / `NetworkError`
  - 其它未归类 → `Unknown`（必须带 `detail` 便于排查）
- `message` 面向用户（可读、可操作）；`detail` 面向日志（技术细节）。
- 前端错误提示按设计规划 §8 的映射表实现，禁止直接把 `detail` 抛给用户。

**示例（Rust）**

```rust
fn read_toc(path: &Path) -> Result<String, AppError> {
    std::fs::read_to_string(path).map_err(|e| AppError {
        code: AppErrorCode::TocParseError,
        message: "无法读取插件描述文件".into(),
        detail: Some(e.to_string()),
        recoverable: true,
    })
}
```

## 4. Rust 编码规范

### 4.1 格式与静态检查
- 统一 `cargo fmt`（默认 rustfmt 配置），提交前必须格式化。
- 必须通过 `cargo clippy --all-targets -- -D warnings`（clippy 告警视为错误）。
- 禁止 `#[allow(...)]` 大范围压制告警；确需豁免要就近注释原因。

### 4.2 命名
- 类型/枚举：`UpperCamelCase`；函数/变量/模块：`snake_case`；常量：`SCREAMING_SNAKE_CASE`。
- domain 结构体字段与 SQLite 列名、TS 字段保持语义一致。
- 序列化统一 `#[serde(rename_all = "snake_case")]`，与前端契约对齐。

### 4.3 结构与错误
- 公共函数返回 `Result<T, AppError>`；内部辅助函数可用具体错误再向上映射。
- 用 `?` 传播错误，不吞异常。
- 异步 Provider 用 `async_trait`；IO 密集操作放 async，CPU 密集（解压/校验）可用 `spawn_blocking`。

### 4.4 文件系统安全
- 所有路径操作走 `infra/fs/path_utils.rs`，不在业务代码里手写路径拼接的平台差异。
- 破坏性操作顺序固定：**校验 → 备份（移动而非删除）→ 执行 → 失败回滚 → 清理**。
- 临时目录用完即清理；清理失败仅记日志，不影响主流程结果。

### 4.5 依赖
- 新增 crate 需在 PR 说明理由；优先成熟库（`serde`、`rusqlite`、`zip`、`reqwest`、`tokio`、`thiserror` 等）。
- 不引入未使用依赖；`Cargo.toml` 保持整洁。

## 5. TypeScript / React 编码规范

### 5.1 格式与静态检查
- 统一 `eslint` + `prettier`，提交前 `pnpm lint` 与 `pnpm format` 通过。
- 必须通过 `tsc --noEmit`（严格类型检查，`strict: true`）。
- 禁止 `any`（确需用 `unknown` 并收窄）；禁止 `// @ts-ignore`（用 `@ts-expect-error` 且注明原因）。

### 5.2 命名与结构
- 组件文件 `PascalCase.tsx`；hooks `useXxx.ts`；工具 `camelCase.ts`。
- 页面放 `pages/`，可复用组件放 `components/`，全局状态放 `stores/`。
- 类型集中在 `types/`，与 Rust domain 一一对应，字段用 `snake_case` 与后端序列化对齐（或在 `services` 层集中转换，二选一并全局统一）。

### 5.3 数据流与边界
- 所有后端调用只经 `services/*Api.ts`，组件内禁止直接 `invoke`。
- 组件不写业务规则（如插件目录判定、路径拼接、`.toc` 解析），只做展示与交互。
- 统一 loading / empty / error 三态；错误按 `AppErrorCode` 展示提示与建议操作。
- 状态管理用 Zustand/Jotai（设计规划指定），避免在组件里散落全局状态。

### 5.4 mock 策略
- `services/` 提供 mock 实现与真实实现，用环境开关（如 `VITE_USE_MOCK`）切换，便于 A8 与后端并行。
- mock 数据结构必须与真实返回一致，切换不改组件代码。

## 6. 数据库规范

- schema 变更只能通过 `infra/db/migrations.rs`，migration 幂等、可重复执行。
- 表结构以设计规划 §9 为准；新增字段走新 migration，不改历史 migration。
- JSON 字段（`*_json`）读写统一封装，默认值为 `'[]'` / `'{}'`。
- 外键关系按设计规划声明；删除父记录前处理子记录。
- DB 文件路径只能来自 `PlatformAdapter.app_data_dir()`，禁止硬编码。

## 7. 测试规范

### 7.1 覆盖要求（对应设计规划 §16）
- 单元测试必须覆盖：`.toc` 解析、插件目录识别、zip 结构识别、path utils、Provider mock、repository、InstallPlan 生成。
- 集成测试必须覆盖：mock WoW 扫描、AddOns 扫描、zip 安装成功、zip 安装失败回滚、WTF 快照创建/恢复、Profile 创建/应用。
- 所有涉及文件系统的模块必须提供 fixture 测试。

### 7.2 组织方式
- Rust 单测就近 `#[cfg(test)] mod tests`；集成测试与 fixture 放 `src-tauri/src/tests/`。
- fixture 目录：`wow_retail_mock/`、`addon_zips/`、`toc_samples/`（可扩展 `wtf_mock/`）。
- 测试用临时目录（`tempfile`），不污染真实系统路径；测试后清理。
- 前端关键交互与错误码映射需有测试。

### 7.3 通过门槛
- 合并前必须：`cargo build`、`cargo test`、`cargo clippy -D warnings`、`tsc --noEmit`、`pnpm lint` 全绿。
- 破坏性操作的「失败回滚」路径必须有对应测试，不能只测 happy path。

## 8. Git 与提交规范

### 8.1 分支
- 主分支：`main`（受保护，禁止直接推）。
- 任务分支命名：`feat/A{n}-{简述}`，如 `feat/A4-installer-pipeline`；修复用 `fix/...`；文档用 `docs/...`。

### 8.2 提交信息（Conventional Commits）
```text
<type>(<scope>): <subject>

type: feat | fix | refactor | test | docs | chore | build
scope: a0..a9 | installer | scanner | db | ui | provider | platform | docs
```
- subject 用中文或英文均可，聚焦「为什么/做了什么」，一句话。
- 一个提交只做一件事；避免混合无关改动。
- 破坏公共接口的提交，subject 前加 `[BREAKING]`。

示例：
```text
feat(a4): 实现 zip 安装流水线与失败回滚
fix(scanner): 修复多 .toc 时同名优先级判定
docs(docs): 补充 A9 打包 checklist
```

### 8.3 PR 要求
- 关联任务编号（T-A{n}），贴出对应「验收标准」勾选情况。
- 说明测试方式与结果（命令 + 结论）。
- 若变更公共接口，标注 `[BREAKING]` 并说明同步了哪些类型。
- 禁止在 PR 里夹带无关文件（构建产物、`node_modules`、本地 DB 等）。
- 未同步 Rust/TS 类型、未过 CI 检查的 PR 不予合并。

### 8.4 安全红线
- 不提交密钥、token、真实用户路径、真实 WTF/SavedVariables 数据。
- 不 `git add -A` 全量提交；按文件精确添加。
- 不擅自 `push --force`、`reset --hard`、改 git config。
- 破坏性 git/文件操作前先与负责人确认。

## 9. 日志规范

- 统一走 `infra/logging`，不用裸 `println!`（面向用户的 CLI 输出除外）。
- 级别：`error`（失败且影响结果）、`warn`（可恢复异常）、`info`（关键节点）、`debug`（排查细节）。
- 破坏性操作（安装/回滚/恢复/删除）在开始与结束各记一条，含关键路径与结果。
- 日志不记敏感信息；日志文件路径在设置页可查（对应 SettingsPage）。

## 10. 提交前自检清单（Checklist）

开发者/Agent 提交前逐条确认：

- [ ] 代码只落在 `manager/wow-addon-manager/`，文档只落在 `manager/docs/`。
- [ ] 所有对外失败返回 `AppError`，无越界 `unwrap/panic`。
- [ ] Rust domain 与 `src/types/` 双向同步。
- [ ] 前端无路径拼接、无直接 `invoke`、无业务规则下沉到组件。
- [ ] 业务层无平台分支；平台差异都在 Platform Adapter。
- [ ] Provider 无安装逻辑；Installer 负责安装/备份/回滚。
- [ ] 破坏性操作有备份与回滚，并写入 `install_history` 或 snapshot metadata。
- [ ] 涉及 fs 的改动带 fixture 测试；失败回滚路径有测试。
- [ ] `cargo fmt/clippy/test`、`tsc --noEmit`、`pnpm lint` 全绿。
- [ ] 提交信息符合 Conventional Commits；接口破坏已标 `[BREAKING]`。
