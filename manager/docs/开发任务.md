## 3. 里程碑

| 里程碑 | 完成标志 | 覆盖任务 |
|---|---|---|
| **M1 可编译骨架** | `pnpm tauri dev` 可启动，前端调用 `health_check` 成功 | A0 |
| **M2 本地能力就绪** | 能识别 WoW 目录、扫描插件、DB 自动初始化、UI 可展示（mock/真实） | A1,A2,A3,A8 |
| **M3 核心业务闭环** | zip 安装+回滚、WTF 快照、Profile 应用、Provider 下载 | A4,A5,A6,A7 |
| **M4 可交付** | 端到端链路通过，Windows/macOS 可打包 | A9 |

**M4 必须打通的最小可验收链路**

```text
启动应用 → 手动选择 WoW 目录 → 后端校验 → 扫描 AddOns → 展示插件列表
→ 选择本地 zip → 生成 InstallPlan → 执行安装（先备份旧插件）→ 刷新列表
→ 创建 WTF 快照 → 创建 Profile → 应用 Profile → 恢复 WTF 快照
```

### T-A0 · 项目骨架与基础契约

- **层次**：全栈脚手架　**依赖**：无　**批次**：1（阻塞全部）

**目标**：创建 `manager/wow-addon-manager/` 下的 Tauri + React + TS + Rust 工程骨架，建立模块目录、领域模型、错误模型与 command 框架。只搭可编译骨架，不实现业务。

**交付物**
- 可启动的 Tauri 应用。
- 前端可调用 `health_check()` 并展示结果。
- Rust domain 类型编译通过，可序列化给前端。

**实现清单**
1. 初始化 `pnpm` + `Vite` + `React` + `TS` 前端工程于 `manager/wow-addon-manager/`。
2. 初始化 `src-tauri`（Cargo 工程、`tauri.conf.json`、`main.rs`/`lib.rs`）。
3. 建立前端目录：`app/ pages/ components/ services/ stores/ types/`（按设计规划 §5）。
4. 建立 Rust 模块骨架：`commands/ app/ domain/ services/ platform/ infra/ scanner/ installer/ providers/ tests/`。
5. 定义 domain models（设计规划 §6）：`GameFlavor / WowInstallation / PermissionCheckResult / LocalAddon / AddonStatus / AddonProviderKind / RemoteAddon / AddonFile / InstallPlan / InstallSource / InstallAction / InstallResult / ConfigSnapshot / SnapshotScope / Profile`。
6. 定义 `AppError` / `AppErrorCode`（设计规划 §8），实现 `serde` 序列化。
7. 建立前端 `types/domain.ts`、`types/command.ts`、`types/errors.ts`，与 Rust 类型一一对应。
8. 实现 `health_check` command + 前端 `tauriClient.ts` 调用封装。
9. 配置基础日志（`infra/logging`）。
10. 配置 SQLite 连接初始化入口（仅入口，不要求完整 schema）。

**显式不做**：插件扫描、安装逻辑、UI 细节、外部 Provider。

**验收标准**
- [ ] `pnpm install && pnpm tauri dev` 可启动应用。
- [ ] 前端页面显示 health check 成功。
- [ ] `cargo build` 通过，domain 类型可序列化。
- [ ] 前端 `types/` 与 Rust domain 字段命名一致（`serde(rename_all="snake_case")` 对齐）。
- [ ] 目录结构与设计规划 §5 一致，全部位于 `manager/wow-addon-manager/`。

**关联 Command & 模型**：`health_check`；全部 domain 模型与 `AppError`。

**测试要求**：`AppError` 序列化单测；前端类型编译通过（`tsc --noEmit`）。

### T-A1 · Platform Adapter 与客户端目录识别

- **层次**：Platform Adapter　**依赖**：A0　**批次**：2（可并行）

**目标**：实现 Windows/macOS 平台适配层，支持自动检测与手动校验 WoW 安装目录，识别 flavor 与 `AddOns`/`WTF` 路径。

**交付物**：可返回本地 WoW installation 列表；可校验用户目录；可识别路径与权限。

**实现清单**
1. 实现 `PlatformAdapter` trait（设计规划 §10.1）。
2. 实现 `WindowsPlatformAdapter`、`MacPlatformAdapter`。
3. 默认路径扫描（§10.2 / §10.3）+ 手动路径校验。
4. flavor 识别：`_retail_ / _classic_ / _classic_era_ / _ptr_`（§10.4）。
5. 校验每个 flavor 的 `Interface/AddOns` 与 `WTF`。
6. 生成 `WowInstallation`，含 `PermissionCheckResult`。
7. 实现权限检测（可读/可写/原因）。
8. 实现 commands：`detect_installations`、`validate_installation_path`、`add_installation`、`list_installations`。

**显式不做**：插件扫描、安装器、完整 UI。

**验收标准**
- [ ] 给定 mock WoW 目录能识别 installation。
- [ ] 无效目录返回 `InvalidInstallationPath`。
- [ ] 无权限目录返回 `PermissionDenied` 或 `permission.writable=false`。
- [ ] 平台判断只存在于 `platform/`，业务层无 `if windows/macos`。
- [ ] `add_installation` / `list_installations` 与 A3 repository 集成（如 A3 未就绪，先接 mock 存储并留 TODO）。

**关联 Command & 模型**：§7.1 全部；`WowInstallation`、`GameFlavor`、`PermissionCheckResult`。

**测试要求**：mock WoW 目录 fixture（`tests/fixtures/wow_retail_mock/`）；无效/无权限目录用例；flavor 识别用例。

### T-A2 · `.toc` 解析与插件扫描

- **层次**：Scanner　**依赖**：A0（A1 最好完成，可用 mock installation 并行）　**批次**：2

**目标**：实现 `Interface/AddOns` 目录扫描与 `.toc` 元数据解析，输出 `LocalAddon`。

**交付物**：给定 `AddOns` 目录返回插件列表，支持依赖与 SavedVariables 字段。

**实现清单**
1. 实现 `TocParser`（§11）：仅解析 `##` 元数据行，key 忽略大小写，value 去空格，逗号字段拆数组。
2. 实现 `AddonFolderDetector`（扫描侧的目录判定，与 A4 zip 侧共用规则可抽公共函数）。
3. 实现 `AddonScanner`：遍历 `Interface/AddOns`。
4. 多 `.toc` 优先级：优先同名 `.toc`，否则第一个可解析。
5. 解析失败标记 `broken`，不中断整个扫描。
6. 支持 `.disabled` 目录识别（与 A6 的启停策略对齐）。
7. 输出 `LocalAddon`（含 `normalized_folder_name`）。
8. 实现 commands：`scan_addons`、`list_addons`。

**显式不做**：Provider、远程更新判断、安装。

**验收标准**
- [ ] 能解析标准 `.toc` 与缺字段 `.toc`。
- [ ] 能处理中文、空格、大小写差异。
- [ ] 目录无 `.toc` 不崩溃，标记 `broken`。
- [ ] 单个插件解析失败不影响其它插件。
- [ ] `list_addons` 与 A3 repository 集成（未就绪时先返回扫描结果并留 TODO）。

**关联 Command & 模型**：§7.2 中 `scan_addons` / `list_addons`；`LocalAddon`、`AddonStatus`、`TocMetadata`。

**测试要求**：`toc_parser` 单测（标准/缺字段/中文/多 toc）；mock `AddOns` 集成测试（`tests/fixtures/toc_samples/`）。

### T-A4 · 安装计划、zip 识别与安全安装器

- **层次**：Installer　**依赖**：A0；集成 A2（`.toc`）、A3（`install_history`/`addons`）　**批次**：3

**目标**：实现本地 zip 安装流水线（计划→备份→替换→回滚），保证失败不破坏旧插件。

**交付物**：可从本地 zip 安装插件；支持多目录插件包；失败自动回滚。

**实现清单**
1. 实现 `ZipService`（校验、解压到临时目录）。
2. 实现 `AddonFolderDetector`（§12.3 识别规则，与 A2 共享）。
3. 实现 `InstallPlanner`：生成 `InstallPlan`（含 `actions`、`backup_path`、`warnings`）。
4. 实现 `InstallExecutor`：备份旧目录→复制新目录→重扫→写 DB→记 `install_history`→清理。
5. 实现 `RollbackManager`：失败时删新目录、恢复旧目录、回滚 DB、记 `failed`。
6. 支持三种 zip 结构（平铺多目录 / 带版本前缀 / `Interface/AddOns` 嵌套）。
7. 实现 commands：`create_install_plan_from_zip`、`execute_install_plan`、`install_addon_from_zip`、`rollback_install`。

**显式不做**：远程 Provider、复杂 UI、CurseForge/Wago。

**验收标准**
- [ ] 标准 zip 可安装。
- [ ] `Interface/AddOns` 嵌套 zip 可安装。
- [ ] 多插件目录 zip 可安装。
- [ ] 无 `.toc` 的 zip 返回 `NoAddonFolderDetected`。
- [ ] 更新已有插件前生成备份。
- [ ] 安装失败可恢复旧目录，且旧插件完好。
- [ ] `install_history` 有成功/失败记录。

**关联 Command & 模型**：§7.3；`InstallPlan / InstallSource / InstallAction / InstallResult / DetectedAddonFolder`。

**测试要求**：zip fixture（`tests/fixtures/addon_zips/`）：标准/多插件/嵌套/无效；安装失败回滚集成测试。

### T-A5 · 配置快照、WTF 备份与恢复

- **层次**：Service　**依赖**：A0、A1、A3　**批次**：3

**目标**：实现 `WTF` 快照创建、恢复、删除与元数据管理，破坏性操作前自动备份、失败自动回滚。

**交付物**：可备份/恢复/删除 `WTF`；快照含 `metadata.json` 与 DB 记录。

**实现清单**
1. 实现 `ConfigService`。
2. 支持 `FullWtf` 快照；`Addon` 范围第一版可简化。
3. 快照压缩为 zip，存储于 `<ApplicationSupport>/snapshots/<installation_id>/<snapshot_id>/`（§13.2）。
4. 写 `metadata.json`（含 `addon_versions`）。
5. 保存 `config_snapshots` DB 记录。
6. 恢复前：校验快照可读 → 备份当前 `WTF`（移动而非覆盖）→ 恢复 → 失败回滚（§13.3）。
7. 实现 commands：`create_config_snapshot`、`list_config_snapshots`、`restore_config_snapshot`、`delete_config_snapshot`。

**显式不做**：Lua diff、云同步、账号/角色级复杂选择（仅保留模型）。

**验收标准**
- [ ] `WTF` 存在时可创建 `FullWtf` 快照。
- [ ] 快照包含 `metadata.json`。
- [ ] DB 记录正确。
- [ ] 恢复前自动创建当前 `WTF` 备份。
- [ ] 恢复失败不破坏当前 `WTF`（自动回滚）。
- [ ] 删除快照同时删文件与 DB 记录。

**关联 Command & 模型**：§7.4；`ConfigSnapshot / SnapshotScope / RestoreResult`。

**测试要求**：mock `WTF` 目录 fixture；恢复失败回滚集成测试。

### T-A7 · Provider 框架与 LocalZip/GitHub Provider

- **层次**：Providers　**依赖**：A0（A4 完成后集成安装）　**批次**：3

**目标**：实现 Provider 抽象，先支持 `LocalZipProvider`、`ManualUrlProvider`、`GitHubReleaseProvider`。Provider 只搜索/取文件/下载，不负责安装。

**交付物**：可扩展 Provider 框架；GitHub release 可作插件来源；Installer 不依赖具体 Provider。

**实现清单**
1. 实现 `AddonProvider` trait（`async_trait`，§14.1）。
2. 实现 `ProviderService`（注册/分发）。
3. 实现 `LocalZipProvider`、`ManualUrlProvider`、`GitHubReleaseProvider`。
4. 下载后交给 A4 的 Installer 执行安装（`install_addon_from_provider` 内部：Provider 下载 → Installer 安装）。
5. Provider 错误统一转 `AppError`（`ProviderError` / `NetworkError`）。
6. 实现 commands：`search_remote_addons`、`get_remote_addon_files`、`install_addon_from_provider`、`check_addon_updates`。
7. `Wago` / `CurseForge` 仅保留枚举与接口，不实现。

**显式不做**：Wago、CurseForge、复杂搜索排名、API Key 管理。

**验收标准**
- [x] Provider trait 可插拔（新增 Provider 不改 Installer）。

## 5. 接口契约与协作约束（摘要）

> 完整约束见 [开发规范.md](./开发规范.md) §模块边界与协作约束。以下为任务执行时最易踩坑的硬约束。

1. **类型双向同步**：改 Rust domain 必须同步 `src/types/`，反之亦然。
2. **错误统一**：任何失败路径必须返回 `AppError`，禁止裸 `String`/`panic` 越过 command 边界。
3. **前端零路径拼接**：所有路径由后端返回，前端只展示。
4. **破坏性操作先备份**：安装、恢复、Profile 应用前必须可备份、可回滚。
5. **Provider 与 Installer 分离**：Provider 只下载，Installer 才安装/写库/备份/回滚。
6. **平台差异只在 Platform Adapter**：业务层禁止平台分支。
7. **文件系统模块必须带 fixture 测试**。
8. **破坏性操作必须落 `install_history` 或 snapshot metadata**。

## 6. 并行开发的接口先行策略

为让第 2/3 批任务真正并行，A0 完成后**接口先冻结**：

- `src-tauri/src/domain/*` 与 `src/types/*`：由 A0 定义并冻结，后续任务只扩展不破坏。
- `src-tauri/src/commands/*`：先建**空实现签名**（返回 `AppError::Unknown` 或 mock），各任务再填充。
- `src/services/*Api.ts`：先建**函数签名 + mock 实现**，A8 据此开发 UI，后端就绪后切真实。
- 跨任务共享逻辑（如 `AddonFolderDetector`、`path_utils`）由先到任务实现并抽公共模块，另一任务复用。

冲突处理：接口变更需在 PR 描述中标注 `[BREAKING]` 并@相关任务负责人；未同步类型的 PR 不予合并。
