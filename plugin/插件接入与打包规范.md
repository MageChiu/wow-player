# 插件接入与打包规范

> 适用范围：`plugin/` 下所有插件与整合包 **WowPlayerSuite**（Hub）。
> 配套文档：整体设计见 [WowPlayerSuite-设计.md](WowPlayerSuite-设计.md)；协作流程见 [chat/开发协作规范.md](chat/开发协作规范.md)。

本规范回答三件事：**插件怎么被 Hub 管理、新增插件怎么做、哪些插件一起打包**。

---

## 1. 两种接入路径（先分清）

"被 Hub 管理" 和 "被打进整合包" 是**两件不同的事**，对应两类插件：

| | 谁安装 | 是否进整合包 | 需要维护 |
|---|---|---|---|
| **整包成员** | 随整合包一起发布 | 是 | `KNOWN`(bundled=true) + `PLUGINS` 两处 |
| **社区/第三方插件** | 用户自行单独安装 | 否 | 仅 `KNOWN`(bundled=false) 一处 |

- 社区插件：Hub 只负责"发现并统一管理"，不打包、不分发。
- 整包成员：既被管理，又随 bundle 一起发布。

**核心原则：接入任何插件都不需要修改那个插件本身。** 适配知识全部集中在 Hub 侧。这样才能纳入"不知道 Hub 存在"的第三方插件。

---

## 2. 接入契约（插件需要满足什么）

Hub 靠 WoW 的公共接口发现并驱动插件，插件只要满足以下**既有惯例**即可被接入，无需为 Hub 添加任何代码：

1. **文件夹名 = `.toc` 名**：如 `MyChat/MyChat.toc`。Hub 用它做 `C_AddOns.IsAddOnLoaded` / `GetAddOnMetadata`。
2. **注册了斜杠命令**：通过全局 `SlashCmdList["KEY"]`（`KEY` 为大写键名，如 `MYCHAT`）。Hub 用它驱动插件。
3. **命令支持"打开设置"参数**：约定为 `config`（即 `/xxx config` 能打开设置面板）。Hub 调 `SlashCmdList["KEY"]("config")` 打开该插件设置。

> 本仓库的 MyChat / RoleManager / CombatCoach / QuickDisenchant 都天然满足以上三条（见各自 `Commands.lua`），因此零改动即可接入。

### 2.1 可选：深度集成（自愿注册）

如果某插件愿意暴露更丰富的能力（如让 Hub 幂等控制其小地图按钮），可在自己 Init 时**自愿**向全局 API 上报，Hub 会合并这些能力（优先于自动发现）：

```lua
-- 可选。没有 Hub 时这段只往一个空壳全局表登记，无副作用。
local api = WowPlayerSuiteAPI or {}
WowPlayerSuiteAPI = api
api._pending = api._pending or {}
local descriptor = {
  title = "YourAddon",
  version = "1.0.0",
  openSettings = function() --[[ 打开你的设置 ]] end,
  setMinimapShown = function(show) --[[ 幂等显隐你的小地图按钮 ]] end,
  getMinimapShown = function() return true end,
}
if type(api.Register) == "function" then api:Register(descriptor)
else api._pending[#api._pending + 1] = descriptor end
```

**这不是接入的必要条件**，仅供想深度集成的插件使用。绝大多数插件（含社区插件）走第 2 节的自动发现即可。

## 3. 单一清单：`Discovery.lua` 的 `KNOWN`

Hub 运行时"认识哪些插件"由 [WowPlayerSuite/Discovery.lua](WowPlayerSuite/Discovery.lua) 的 `KNOWN` 表决定，这是**运行时管理的单一源**。每条字段：

| 字段 | 含义 | 示例 |
|---|---|---|
| `addon` | 插件文件夹名（= toc 名） | `"MyChat"` |
| `slash` | `SlashCmdList` 里的键名（大写） | `"MYCHAT"` |
| `openArg` | 主操作参数：有设置面板用 `"config"`（按钮"打开设置"）；空命令即开主窗口的用 `""`（按钮"打开"） | `"config"` / `""` |
| `category` | 分类，**必须等于该插件所在的一级目录名**（见第 3.1 节） | `"chat"` |
| `displaySlash` | 概览页展示的命令 | `"/mychat"` |
| `bundled` | 是否为整包成员（true/省略） | `true` |

版本号**不在此维护**——Hub 用 `C_AddOns.GetAddOnMetadata(addon, "Version")` 从各插件 toc 实时读取，避免清单与实际版本脱节。

### 3.1 分类 = 一级目录（目录即分类）

**约定：`plugin/` 下的每个一级目录 = 一个分类。** 映射定义在 [WowPlayerSuite/Constants.lua](WowPlayerSuite/Constants.lua) 的 `C.CATEGORY`：

| 一级目录 | 分类显示名 |
|---|---|
| `chat` | 聊天增强 |
| `role-manager` | 角色数据 |
| `enhancement` | 战斗增强 |
| `trade` | 物品与交易 |
| `mini-game` | 休闲小游戏 |

概览页据此**分组显示**。`KNOWN` 每条的 `category` 必须等于该插件源目录的一级目录名，`package-suite.sh` 会**交叉校验**（写错报错）。新增分类：在 `C.CATEGORY` 与 `C.CATEGORY_ORDER` 各加一行。

## 4. 新增插件的标准步骤

### 4.1 新增「社区/第三方插件」（仅管理，不打包）

1. 确认该插件满足第 2 节接入契约（文件夹名、`SlashCmdList` 键、`config` 参数）。
   - 若拿不准键名，可在游戏内看该插件文档，或用 `/dump SlashCmdList` 之类方式确认。
2. 在 `Discovery.lua` 的 `KNOWN` 加一行，**`bundled` 省略或置 `false`**：
   ```lua
   { addon = "SomeAddon", slash = "SOMEADDON", settings = "config", displaySlash = "/sa" },
   ```
3. 完成。无需改动那个插件，无需动打包脚本。

> 若该插件的"打开设置"参数不是 `config`（比如是 `options`），把 `settings` 改成对应值即可。

### 4.2 新增「整包成员」（既管理又打包）

1. 同 4.1 第 1 步确认接入契约。
2. 在 `Discovery.lua` 的 `KNOWN` 加一行，**`bundled = true`**。
3. 在 [package-suite.sh](package-suite.sh) 的 `PLUGINS` 数组加一行 `"插件名|相对 plugin 的源目录"`：
   ```bash
   "SomeAddon|category/SomeAddon"
   ```
4. 运行 `./package-suite.sh --check`，确认「整包成员与 Discovery.KNOWN 一致校验通过」。
   - 脚本会**交叉核对**两处清单，漏改任何一处都会报错，防止不同步。

## 5. 打包机制

### 5.1 两种打包并存

| 方式 | 脚本 | 产出 | 适用 |
|---|---|---|---|
| **单插件** | 各插件的 `package.sh` | `<插件>-<版本>.zip` | 只想要某一个插件；社区分发 |
| **整合包** | `plugin/package-suite.sh` | `WowPlayerSuite-Bundle-<套件版本>.zip` | 一次装齐全部整包成员 |

两者互不排斥：整合包用户下 bundle，单插件用户下对应 zip。产出统一落在 [plugin/dist/](dist)。

### 5.2 整合包内容与安装

`package-suite.sh` 把各整包成员汇集进**一个 zip**，解压后为并列的多个插件文件夹：

```
WowPlayerSuite-Bundle-0.1.0.zip 解压后:
├── WowPlayerSuite/    ← Hub
├── MyChat/
├── RoleManager/
├── CombatCoach/
└── QuickDisenchant/
```

整个拖进 `Interface/AddOns/` 即全部装齐。打包时自动剔除 `package.sh`、`.DS_Store`、`.git`、`*.swp`、`*.bak` 等开发/系统文件。

### 5.3 版本号与升级

- **套件版本号** `SUITE_VERSION` 定义在 `package-suite.sh` 顶部，独立于各插件版本。发布整合包时递增这一处。
- 各插件版本仍遵循 [chat/开发协作规范.md](chat/开发协作规范.md) 的语义化版本约定（toc 的 `## Version` 与 `Constants.lua` 的 `C.VERSION` 必须一致，单插件 `package.sh` 会校验）。
- **升级方式**：重新下载 bundle 覆盖 `AddOns/` 下对应文件夹即可。WoW 插件**无法游戏内自动下载更新**，均为手动覆盖。

## 7. 一致性检查清单（改动后自检）

- [ ] 新增插件是否已加入 `Discovery.lua` 的 `KNOWN`？
- [ ] 若为整包成员，是否已加入 `package-suite.sh` 的 `PLUGINS`？
- [ ] `./package-suite.sh --check` 是否通过（含交叉核对）？
- [ ] 若涉及新增/移除整包成员，是否更新了本文件第 6 节清单？
- [ ] 发布整合包时，是否递增了 `SUITE_VERSION`？
