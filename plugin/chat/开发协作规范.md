# MyChat 开发协作规范

> 核心原则：**先聊清楚，再动手。任何开发在未达成一致前，不写代码。**
>
> 本规范用于约束 MyChat 插件（`plugin/chat/`）的所有开发协作。目的是避免"问题没聊透就急着写、反而绕远"，把力气花在正确的方向上。

---

## 五阶段流程

每个阶段有明确产出，**用户确认后才进入下一阶段**。禁止跨阶段抢跑。

### 1. 澄清问题（Clarify）
- 先复述对问题的理解：**具体现象是什么、怎么复现、期望结果是什么**。
- 有歧义先问，**绝不猜用户意图**。
- 反例：用户只想"上下键能翻历史、能重发"，却自行发散到 `/6` 预填、编辑框 reparent 等无关细节 —— 这是跑偏。

### 2. 定位根因（Diagnose）
- 只读代码 + 查证官方文档/源码，给出根因**和证据**。
- **不凭记忆猜 API**。历史教训：`-t`、`category.ID`、`C_Club.GetStreamInfo` 都曾凭记忆猜错。拿不准就查 Wiki / townlong-yak 源码。
- 根因未确定时，明确标注"待验证"，不当作结论抛出。

### 3. 确定方案（Design）
- 给出方案 + 取舍 + 影响范围；**需要用户拍板的点明确列出**。
- **用户确认方案后，才进入实现。**

### 4. 实现（Implement）——仅在 1–3 全部确认后
- 严格按确认的方案改，**不擅自扩大范围**（不顺手重构、不加没要求的功能）。
- 守住既有设计原则（见下）。

### 5. 交付验证（Verify）
- 打包 + 说明部署步骤 + 用户实测；**用户确认通过才算完成**。

### 例外
- 仅当用户明确表示"这个直接改吧 / 小问题直接动手"时，才可跳过 1–3 直接实现。

---

## 贯穿始终的设计原则

这些原则在任何阶段都不可违背：

1. **增强而不接管**：只增强 12.0 原生聊天，绝不替换/覆写聊天框。关掉插件后原生聊天完全不受影响。
2. **不重复原生已有功能**：12.0 原生已做的（时间戳、频道名、玩家名职业色等）一律不重复实现 —— 重复会导致"显示两遍"或破坏链接。
3. **可降级**：所有增强都能被关闭；出问题可 `/mychat safe` 进安全模式或直接停用插件。
4. **错误隔离**：所有回调、事件处理用 `pcall`（`Core:SafeCall`）包裹，单点失败不拖垮整个插件。
5. **配置必须真实生效**：设置面板里的每一项都要有代码真正读取并生效，**禁止空壳开关**。
6. **战斗安全**：涉及发送/受保护操作的动作走 `CombatGuard`，战斗中延迟到脱战执行。

## 补充约定

- **打包与版本号**：每次改动后重新打包（`./package.sh --output ../dist`）供验证，但**打包 ≠ 涨号**。
  - **仅在"交付"时递增版本号**——即这一批功能/修复已稳定、要正式交给用户实测或部署时才涨一次。开发中途的反复修改、连续 bug 修复沿用同一版本号，重新打包即可。
  - **遵循语义化版本 `MAJOR.MINOR.PATCH`**：
    - `PATCH`：bug 修复、文案/样式微调（如 `0.1.0 → 0.1.1`）。
    - `MINOR`：新增功能、新面板/新命令等向后兼容的能力（如 `0.1.x → 0.2.0`）。
    - `MAJOR`：大重构或不兼容改动（如存档结构变更、移除命令）。发布前的 `0.x` 阶段可暂不启用。
  - 涨号时 `.toc` 的 `## Version` 与 `Constants.lua` 的 `C.VERSION` 必须同步（打包脚本会校验一致）。
- 涉及 API 行为时，优先引用官方来源（Warcraft Wiki、townlong-yak FrameXML 源码）作为证据。
- 提交/打 tag / 推送 GitHub 等对外可见操作，**先征得用户同意**再执行。

> 注：本"打包与版本号"约定适用于仓库下所有插件（`plugin/` 各子目录，如 `chat/`、`role-manager/`），不限于 MyChat。
